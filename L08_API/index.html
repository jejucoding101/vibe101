API가 뭐예요?

API는 Application Programming Interface의 약자입니다. 직역하면 "프로그램끼리 소통하는 약속된 방법"입니다.

어렵게 들리지만, 사실 우리 일상 속에 넘쳐납니다. 식당에서 웨이터에게 주문하면 주방에서 음식이 나오는 것처럼, API는 여러분의 프로그램(손님)과 서버(주방)를 연결해주는 웨이터입니다.

🍽️ 식당 비유로 이해하기
👤
손님 (앱)
"비빔밥 주세요!"
→ 요청(Request)
🤵
웨이터 (API)
주문을 전달하고
음식을 가져다줌
👨‍🍳
주방 (서버)
"비빔밥 나왔습니다!"
→ 응답(Response)
💡
손님은 주방에 직접 들어갈 필요가 없습니다. 웨이터(API)가 약속된 방식(메뉴판)으로 주문을 받고, 결과(음식)를 가져다줍니다. 프로그래밍에서도 마찬가지로, 여러분은 서버 내부를 몰라도 API를 통해 원하는 데이터를 얻을 수 있습니다.

왜 API를 사용하나요?

세상의 모든 기능을 여러분이 직접 만들 필요가 없습니다. 이미 잘 만들어진 서비스의 기능을 빌려 쓰는 것이 API의 핵심 가치입니다.

상황API 없이 😱API 사용 ✅
날씨 앱기상 관측소를 직접 운영해야 함기상청 API 한 줄 호출
지도 앱지도 데이터를 직접 수집·업데이트Google Maps API 활용
로그인보안 시스템 전체를 직접 구현Google/카카오 OAuth API
번역AI 모델을 직접 학습시켜야 함DeepL/Papago API 호출
결제PG사와 직접 연동 개발토스페이먼츠 API 연동
🧩
핵심: API는 "남이 만든 강력한 기능을 내 프로그램에 레고 블록처럼 붙여 쓰는 것"입니다. 마치 레고 블록을 조합해서 원하는 작품을 만드는 것과 같습니다.

API의 핵심: 요청(Request)과 응답(Response)

API의 모든 동작은 딱 2가지로 이루어집니다. 요청(Request)을 보내고, 응답(Response)을 받는 것입니다. 이것만 이해하면 API의 절반은 끝입니다.

📤 요청 (Request)
메서드GET
주소/weather?city=Seoul
헤더Accept: application/json
의미"서울 날씨 알려줘!"
→ ←
📥 응답 (Response)
상태200 OK
형식application/json
데이터{"city":"Seoul","temp":18}
의미"서울은 18도야!"

API 주소(URL) 해부하기

API를 호출하려면 정확한 주소(URL)를 알아야 합니다. URL은 여러 부분으로 나뉘며, 각각 역할이 다릅니다.

https://api.weather.example.com/v1/current?city=Seoul&lang=ko
프로토콜 (통신 방식)
호스트 (서버 주소)
버전 (API 버전)
경로 (어떤 데이터)
쿼리 파라미터 (조건)
구성요소역할예시
프로토콜데이터를 주고받는 통신 규약. HTTPS는 암호화된 안전한 연결입니다.https://
호스트API 서버의 집 주소. 어떤 서버에 요청할지 결정합니다.api.weather.example.com
버전API의 버전 번호. 서비스가 업데이트되어도 기존 코드가 깨지지 않게 합니다./v1
경로(Path)어떤 종류의 데이터를 원하는지 지정합니다./current
쿼리 파라미터세부 조건을 전달합니다. &로 여러 조건을 연결합니다.city=Seoul&lang=ko

REST API란?

REST(Representational State Transfer)는 API를 설계하는 가장 대중적인 규칙입니다. 간단히 말하면, "인터넷 주소(URL)와 HTTP 메서드를 조합해서 데이터를 요청하는 방법"입니다.

REST API를 이해하는 열쇠는 자원(Resource)이라는 개념입니다. 서버에 있는 모든 데이터는 자원이고, 각 자원에게는 고유한 URL 주소가 있습니다. 마치 도서관의 모든 책에 고유 번호(청구기호)가 있는 것과 같습니다.

HTTP 메서드 — 4가지 기본 동작 (CRUD)

REST API에서는 무엇을 할 것인가를 HTTP 메서드로 표현합니다. 핵심 4가지를 CRUD라고 부릅니다.

GET
조회 (Read)
데이터를 가져옵니다. 서버의 데이터를 변경하지 않습니다.
GET /users → 전체 사용자 목록
GET /users/1 → 1번 사용자 정보
POST
생성 (Create)
새로운 데이터를 만듭니다. 요청 본문(body)에 데이터를 담아 보냅니다.
POST /users
Body: {"name":"철수","age":14}
PUT
전체 수정 (Update)
기존 데이터를 완전히 새 데이터로 교체합니다.
PUT /users/1
Body: {"name":"영희","age":15}
DELETE
삭제 (Delete)
데이터를 삭제합니다. 보통 본문(body) 없이 URL만으로 요청합니다.
DELETE /users/1 → 1번 사용자 삭제
PATCH
부분 수정 (Partial Update)
데이터의 일부분만 수정합니다. PUT과 달리 변경할 필드만 보냅니다.
PATCH /users/1
Body: {"age":16}
🤔
PUT vs PATCH 차이점: PUT은 "문서 전체를 새걸로 교체"하는 것이고, PATCH는 "문서의 특정 부분만 수정"하는 것입니다. 마치 숙제를 다시 쓰는 것(PUT)과 오타만 고치는 것(PATCH)의 차이입니다.

HTTP 상태 코드 — API의 신호등

서버는 응답할 때 반드시 숫자 코드를 함께 보냅니다. 이 코드를 보면 요청이 성공했는지, 어떤 문제가 있는지 바로 알 수 있습니다.

200
OK — 성공!
🍽️ 주문한 음식이 나왔어요
201
Created — 생성 완료
📝 새 메뉴가 등록됐어요
301
Moved — 주소 변경
🏠 식당이 이사했어요
400
Bad Request — 잘못된 요청
📋 메뉴판에 없는 걸 주문
401
Unauthorized — 인증 필요
🔒 회원증이 없어요
403
Forbidden — 접근 금지
🚫 VIP 전용 구역이에요
404
Not Found — 찾을 수 없음
🔍 그 메뉴는 없어요
429
Too Many Requests — 요청 초과
⏳ 주문이 너무 많아요, 잠시만요
500
Internal Server Error
🔥 주방에 불이 났어요
503
Service Unavailable
🔧 오늘은 정기 휴무일이에요
📊
외우는 비법: 2xx = 성공 🎉, 3xx = 이동/리다이렉트 ➡️, 4xx = 내 잘못 (클라이언트 오류) 😅, 5xx = 서버 잘못 (서버 오류) 💥

API의 종류

🌍
공개 API (Open)
누구나 무료로 사용 가능. 가입이나 키 없이 바로 호출할 수 있습니다.
예: JSONPlaceholder, Open-Meteo, PokeAPI
🔑
키 필요 API
가입 후 API 키를 발급받아야 사용 가능. 대부분 무료 요금제가 있습니다.
예: OpenWeather, Google Maps, 공공데이터포털
🏢
비공개 API (Private)
회사 내부에서만 사용. 외부에 공개되지 않은 내부 서비스 연동용입니다.
예: 회사 내부 CRM, ERP 시스템
🔑
API 키란? 도서관 회원증처럼, "이 사용자는 허가된 사용자입니다"를 증명하는 비밀 코드입니다. API 키는 절대 공개 저장소에 올리면 안 됩니다! 누군가 악용하면 비용이 청구될 수 있습니다.

API 응답 데이터 형식: JSON

현대 API의 약 90% 이상이 JSON 형식으로 데이터를 주고받습니다. 7강에서 배운 내용이 여기서 빛납니다.

  api-response.json 
{
  "status": "success",
  "data": {
    "city": "Seoul",
    "temperature": 18.5,
    "unit": "celsius",
    "humidity": 62,
    "wind": {
      "speed": 3.2,
      "direction": "NW"
    },
    "forecast": ["맑음", "구름 조금"]
  },
  "timestamp": "2026-04-22T12:00:00Z"
}
🔍 응답 구조 분석
1
status — 요청 성공 여부를 나타내는 필드
2
data — 실제 데이터가 담긴 핵심 객체
3
data.wind — 중첩 객체로 관련 데이터를 그룹화
4
data.forecast — 배열로 여러 값을 순서대로
5
timestamp — 데이터 생성 시각 (ISO 8601 형식)

API 동작 전체 흐름

여러분이 앱에서 버튼 하나를 누르면, 뒤에서 이런 일이 벌어집니다.

1단계: 사용자가 행동
앱에서 "서울 날씨 보기" 버튼을 클릭합니다.
2단계: 앱이 API에 요청(Request) 전송
GET /weather?city=Seoul 요청을 API 서버에 보냅니다.
3단계: 서버가 요청을 처리
서버는 요청을 확인하고, 데이터베이스나 외부 시스템에서 서울의 날씨 데이터를 조회합니다.
4단계: 서버가 응답(Response) 반환
상태 코드 200 OK와 함께 JSON 형식의 날씨 데이터를 돌려보냅니다.
5단계: 앱이 화면에 표시
앱은 받은 JSON 데이터를 분석(파싱)하여 "서울 18°C ☀️"라고 화면에 보여줍니다.

API 실습 도구

⏱ 10분

API를 배울 때 가장 좋은 방법은 직접 요청을 보내보는 것입니다. 코드를 작성하기 전에, 먼저 이 도구들로 API가 어떻게 동작하는지 눈으로 확인해보세요.

🌐
웹 브라우저
설치 불필요 · 가장 간단
주소창에 API URL을 직접 입력하면 됩니다. GET 요청만 가능하지만, API를 처음 테스트할 때 가장 빠른 방법입니다.
  • 아무것도 설치할 필요 없음
  • API 응답을 바로 눈으로 확인
  • JSON Formatter 확장 설치 시 더 편리함
지금 바로 테스트 →
🦋
Hoppscotch
웹 기반 · 무료 · 오픈소스
브라우저에서 바로 사용하는 API 테스트 도구입니다. 설치가 필요 없고, 직관적인 인터페이스가 장점입니다. 초보자에게 가장 추천합니다.
  • 설치 없이 웹에서 바로 사용
  • GET/POST/PUT/DELETE 모두 지원
  • 응답을 보기 좋게 자동 포맷
  • 무료 · 오픈소스
hoppscotch.io →
🚀
Postman
데스크탑 앱 · 무료 + 유료
업계 표준 API 테스트 도구입니다. 요청 저장, 환경 변수, 테스트 자동화 등 전문적인 기능을 제공합니다.
  • 요청을 컬렉션으로 저장·관리
  • 환경 변수로 API 키 관리
  • 자동 테스트 스크립트 작성 가능
  • 팀 협업 기능
다운로드 →
⌨️
cURL
터미널 명령어 · 내장
터미널(명령 프롬프트)에서 API를 호출하는 명령어입니다. 대부분의 운영체제에 기본 설치되어 있습니다.
  • 추가 설치 필요 없음 (대부분 내장)
  • 스크립트로 자동화 가능
  • 가장 가벼움
📝
VS Code REST Client
확장 프로그램 · 무료
VS Code에서 .http 파일을 만들어 API를 테스트합니다. 코드 에디터를 떠나지 않고 바로 테스트할 수 있어 개발자에게 인기입니다.
  • 코드와 같은 화면에서 테스트
  • .http 파일로 요청 관리
  • Git으로 요청 히스토리 관리 가능
🌙
Insomnia
데스크탑 앱 · 무료 + 유료
Postman의 대안으로, 깔끔한 UI와 GraphQL 지원이 장점입니다. REST와 GraphQL API를 모두 테스트할 수 있습니다.
  • 깔끔하고 직관적인 인터페이스
  • GraphQL 네이티브 지원
  • 환경 변수 및 플러그인 시스템
다운로드 →
🎯
초보자 추천 순서: ① 브라우저에서 URL 직접 입력 → ② Hoppscotch로 다양한 메서드 테스트 → ③ 코드로 직접 호출(fetch/axios). 처음부터 Postman을 쓸 필요 없습니다!

cURL 기본 사용법

# 터미널에서 API 호출 (GET 요청)
curl https://jsonplaceholder.typicode.com/posts/1

# 보기 좋게 출력하기 (Windows PowerShell)
curl https://jsonplaceholder.typicode.com/posts/1 | ConvertFrom-Json

# 보기 좋게 출력하기 (Mac/Linux)
curl https://jsonplaceholder.typicode.com/posts/1 | python -m json.tool
# POST 요청으로 데이터 보내기
curl -X POST https://jsonplaceholder.typicode.com/posts \
  -H "Content-Type: application/json" \
  -d '{"title":"내 첫 포스트","body":"API 배우는 중!","userId":1}'
# 인증 헤더 포함 요청
curl -H "Authorization: Bearer YOUR_API_KEY" \
     -H "Accept: application/json" \
     https://api.example.com/data

# 응답 헤더도 함께 보기
curl -i https://jsonplaceholder.typicode.com/posts/1

이번 강의 체크리스트

  • API가 무엇인지 식당 비유로 설명할 수 있다
  • 요청(Request)과 응답(Response)의 개념을 이해했다
  • URL의 구조(프로토콜, 호스트, 경로, 파라미터)를 읽을 수 있다
  • REST API의 CRUD 4가지 메서드(GET/POST/PUT/DELETE)를 이해했다
  • HTTP 상태 코드(200/400/401/404/500)의 의미를 안다
  • API 실습 도구(Hoppscotch, Postman 등)를 알고 있다
  • 공개 API를 호출해서 JSON 데이터를 받아올 수 있다
  • 받아온 JSON 데이터에서 원하는 값을 추출할 수 있다
